# Project Process

## Prime Rule

Before continuing meaningful work, update the project state:

1. `docs/CURRENT_TASK.md`
2. `docs/CURRENT_THINKING.md`
3. `docs/NEXT_TASK.md`
4. `docs/BACKLOG.md`
5. `docs/BUGS_AND_RISKS.md`
6. `docs/TECH_DEBT.md`
7. `docs/conversation-archive/` when the conversation changed project direction or decisions

This keeps the project portable across humans and AI agents.

## Work Package Rules

Every coding task must be defined as a work package before implementation.

Each work package needs:

- Goal
- Requirement IDs
- Owned modules/files
- Non-owned modules/files that must not be edited
- Acceptance criteria
- Test plan
- Migration plan if data changes
- Security/privacy review notes
- Rollback plan
- Handoff notes

## Short-Burst Commit Rule

Commit in small semantic bursts after each stable mini-milestone. A good commit should have one clear reason to exist, such as one module scaffold, one contract group, one domain rule plus tests, one bug fix, or one planning-doc update.

Use `docs/COMMIT_POLICY.md` and `.gitmessage.txt` for commit format. AI-authored commits must include:

- `AI-Agent`
- `AI-Work-Mode`
- `AI-Commit-Time`
- `Work-Package`
- `Requirements`
- `Process-Docs-Updated`

Do not make a large mixed commit when smaller commits would preserve reviewability and handoff context. Do not commit broken code unless it is an intentional private WIP checkpoint with the recovery path written in `docs/NEXT_TASK.md`.

## Adversarial Thinking Rule

For each major decision, write the best argument against the current plan. A decision is not strong until it survives a serious challenge.

Use ADRs for decisions involving:

- Mobile framework
- Database
- Map provider
- Auth provider
- AI provider
- Scoring formula
- Storage architecture
- Privacy model
- Moderation model
- Repo structure

## File Size And Module Rules

- Target 200-300 lines per source file.
- Split by responsibility before a file becomes hard to test.
- Keep public interfaces small and explicit.
- Avoid cross-module imports that bypass domain boundaries.
- Add local module READMEs when a module becomes non-trivial.
- Prefer many focused tests over one broad test file.

Larger files are allowed only when:

- Generated by a tool.
- Required by framework conventions.
- Documented in `docs/TECH_DEBT.md`.
- Paired with a plan to split later.

## Code Context Comments

Use comments to preserve important context, not to repeat code.

Good comments:

- Reference requirement IDs.
- Reference ADR IDs.
- Explain a scoring or privacy rule.
- Explain why a security check exists.
- Mark temporary compromises with a debt ID.

Bad comments:

- Explain obvious syntax.
- Narrate every line.
- Hide uncertainty without a ticket or ADR.

Example:

```ts
// REQ FR-SCORE-003, ADR-004: zoo captures save to collection but skip normal leaderboard scoring.
```

## Quality Gates

No feature is complete until:

- Requirement IDs are linked.
- Unit tests pass.
- Integration tests pass for changed flows.
- Security and privacy checklist is reviewed.
- Performance risk is considered.
- State docs are updated.
- Next task is written.

## Testing Architecture Rule

Before coding a module, read `docs/qa/README.md` and the relevant QA spec. Add or update test IDs before implementing behavior if the current spec does not describe the expected tests.

Use `docs/qa/TEST_CASE_CATALOGUE.md` as the source for concrete test IDs and `docs/qa/BDD_ACCEPTANCE_SCENARIOS.md` for user-visible acceptance flows.

Launch-blocking areas include privacy DTOs, score state, duplicate/zoo checks, upload idempotency, moderation/report/block flows, and exact-location leak prevention. If these tests fail, stop feature exposure until the contract or implementation is fixed.

Use `docs/qa/LOCAL_PR_CHECKLIST.md` before commits/PRs and `docs/qa/ARCHITECTURE_FITNESS_RULES.md` when adding module boundaries. Test harness structure follows `docs/qa/TEST_HARNESS_ARCHITECTURE.md`; coverage and flaky-test decisions follow `docs/qa/COVERAGE_AND_FLAKY_POLICY.md`.

## AI Agent Handoff Rules

When handing work to another AI agent:

- Point it to `docs/CURRENT_TASK.md` first.
- Point it to `docs/NEXT_TASK.md` second.
- Point it to `docs/conversation-archive/` when original user prompts or long-form decisions matter.
- Give owned files and forbidden files.
- Ask it to update state docs before final response.
- Ask it to preserve the latest visible user prompts/responses as raw archive text or a session summary.
- Ask it to list changed files and tests run.
- Do not ask two agents to edit the same files in parallel.

## End Of Task Checklist

Before ending any task:

- Update current task status.
- Update current thinking with new decisions.
- Add new backlog items.
- Add discovered bugs or risks.
- Add technical debt.
- Update next task.
- Run relevant tests or explain why not.
- Summarize changed files.
